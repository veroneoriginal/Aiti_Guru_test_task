Тестовое задание.

1. Спроектировать и реализовать сервис работы с платежами по заказу

✅ - В сервисе находится список из одного или более заказов, работа с которыми и формирование новых не требуется.
Каждый заказ содержит поля "сумма заказа" и "статус оплаты" (не оплачен, частично оплачен, оплачен).
# app/models/order.py - модель Order

✅ - Для каждого неоплаченного заказа должна быть возможность оплатить заказ одним или несколькими платежами.
# app/services/payment_service.py - функция def deposit, проверяет available_to_pay(), не даёт переплатить.

✅ - Каждый платеж должен иметь тип (наличные, эквайринг).
# app/models/base.py - PaymentType

✅ - У модели платежа должны быть реализованы две операции: депозит и возврат, которые должны в том числе согласованно менять статус оплаты заказа.
- Названия полей и их типы — на усмотрение кандидата.
# app/services/payment_service.py - функции deposit и refund


✅ - Сумма всех платежей по заказу не должна превышать сумму всего заказа.
# app/services/payment_service.py - Проверка amount > available в deposit(), плюс available_to_pay() считает остаток.

✅ - Модели платежей разных типов не должны отличаться друг от друга, т.е. работать с платежами разных типов должно быть возможно не задумываясь об их конкретном типе там, где это не важно.
# app/models/payment.py - Одна модель Payment для всех типов (Single Table).
Поля bank_* nullable для наличных. Работать с платежом можно одинаково, не зная его тип — mark_completed(), amount, status — всё едино.


2. API банка

- Для платежа через банк нужно предложить архитектурное решение, позволяющее:

  ✅ - вызывать API банка для создания платежа;
  # app/infrastructure/bank/client.py.
  Метод acquiring_start() отправляет POST с order_id и amount, получает bank_payment_id.
  Используется в deposit() и refund() для эквайринговых платежей.

  ✅ - хранить состояние этого платежа;
  # app/models/payment.py Реализовано в модели Payment. Три поля: bank_payment_id, bank_status, bank_paid_at.

  ✅ - согласовывать его состояние в нашем приложении и на стороне банка.
  # sync_scheduler.py

Это API не нужно никак реализовывать, реализовать только запросы к нему, руководствуясь нижеуказанными правилами.

Состоит из следующих ресурсов:

 2.1. bank.api/acquiring_start
# app/infrastructure/bank/client.py acquiring_start
✅ - Принимает: JSON с номером заказа, суммой заказа.
✅ - Возвращает:
  - в случае успеха — уникальный ID платежа в банке;
  - в случае неудачи — строку с ошибкой.

# app/infrastructure/bank/client.py acquiring_check
2.2. bank.api/acquiring_check

✅ - Принимает: уникальный ID платежа в банке.
✅ - Возвращает: ID платежа в банке, сумму, статус, дату и время оплаты, либо ошибку «платеж не найден».

Требования

- Написать код обращения к этому API с обработкой ошибок. Учесть, что API банка это внешняя система.
✅ Если банк недоступен» — sync_payment_with_bank внутри вызывает bank_client.acquiring_check(), который бросает BankUnavailableError при таймауте или сетевой ошибке.
И есть обработчик в handlers.py, который вернёт клиенту ответ с кодом 502 и сообщением «API банка недоступен».
Плюс ловятся TimeoutException, RequestError, проверяется статус-код, бросаются BankUnavailableError и BankPaymentNotFoundError.

- Использовать этот код в операциях работы с платежами по заказу.

- Учитывать, что состояние платежей в банке может быть изменено на стороне банка неожиданно для нашего приложения.
✅ Покрыто фоновым циклом синхронизации. Каждые 60 секунд опрашиваем банк по pending-платежам.

✅ - Конкретные названия полей произвольные, типы данных на усмотрение кандидата.


3. Состояния, операции и REST API

✅ - Реализовать операции создания и возврата платежа по заказу с использованием сущностей заказа, платежа и других (при их наличии).
В # app/services/payment_service.py функции deposit() и refund()

✅ - Для этих операций реализовать RESTful-ручки с использованием любого Python веб-фреймворка на усмотрение кандидата.
FastAPI, три ручки в # app/api/routers/payments.py:
POST /payments — создание платежа,
POST /payments/{id}/refund — возврат,
POST /payments/{id}/sync — синхронизация с банком
В # app/api/routers/orders.py - GET /orders/{id}

- В коде ручек нужно:
✅ - проверять входные данные от пользователя;
Pydantic-схемы PaymentCreateRequest и RefundCreateRequest с валидацией:
amount обязательно > 0, max_digits=12, decimal_places=2, order_id — UUID, type — только из enum.

✅ - обрабатывать ошибки внутренних операций с корректным формированием ответа.
Все кастомные исключения зарегистрированы в # app/api/exceptions/handlers.py с правильными HTTP-кодами (400, 404, 502).

✅ - Учитывать, что некоторые операции кандидату будет необходимо вызывать не только из RESTful-ручек.
Бизнес-логика живёт в # app/services/payment_service.py, а не в роутерах.
Функции deposit() и refund() принимают session как аргумент — их можно вызвать откуда угодно:
из ручки, из фоновой задачи, из теста.
sync_payment_with_bank в # app/infrastructure/bank/sync.py тоже вызывается и из ручки /sync, и из фонового sync_scheduler.
